# Метод Pattern Layer

## 1. Назначение
Pattern layer строит «эталонный» диагноз-специфичный паттерн опроса и анамнеза на основе ретроспективных обезличенных документов.
Паттерн затем используется при анализе нового текста для поиска пропусков в сборе жалоб/анамнеза.

## 2. Где реализовано
- `src/analysis/diagnosisPatternService.js`
- использование в анализе: `src/analysis/caseAnalysisService.js`
- хранение: таблица `diagnosis_patterns` в `src/storage/sqliteRepository.js`

## 3. Ключевые сущности
- `diagnosisCodes`: список ICD-кодов.
- `diagnosisKey`: детерминированный ключ (`K74.6+K76.0`).
- `pattern`: JSON-структура с клиническими ожиданиями.

Текущий JSON-контракт `pattern`:
- `required_questions: string[]`
- `red_flags: string[]`
- `common_context: string[]`
- `diagnostic_focus: string[]`

## 4. Алгоритм построения паттерна

## 4.1. Формирование когорты
1. Берутся документы из SQLite по пересечению `diagnosisCodes`.
2. Ограничение выборки: до 120 документов на генерацию.
3. Минимальный порог: `minDocs` (по умолчанию 8).

Если документов меньше порога:
- предупреждение в лог,
- паттерн не считается надежным,
- возврат `null` (или fallback при необходимости).

## 4.2. Генерация паттерна через LLM
1. Формируется промпт с:
- диагнозами,
- набором примеров документов,
- требованием вернуть строго JSON.
2. Вызов через OpenRouter `/chat/completions` с `OPENROUTER_ANALYSIS_MODEL`.
3. Результат парсится как JSON.

Зачем LLM на этом этапе:
- извлекает повторяющиеся клинически значимые паттерны из свободного текста,
- формирует компактный checklist без ручного кодирования под каждый диагноз.

## 4.3. Fallback при ошибке LLM
Если LLM недоступна/вернула невалидный результат:
- используется эвристический паттерн,
- `common_context` строится через частотные токены,
- `required_questions` и `red_flags` заполняются безопасным базовым набором,
- модель помечается как `heuristic-fallback`.

Зачем:
- избегаем «немого» падения,
- pipeline остается рабочим в production.

## 4.4. Сохранение и повторное использование
Паттерн сохраняется по `diagnosisKey` в `diagnosis_patterns`.
При анализе нового кейса:
1. сначала читается сохраненный паттерн,
2. если паттерна нет, сервис пробует построить его «на лету».

## 5. Как паттерн используется в анализе нового кейса
Pattern layer не выдает финальный вердикт сам.
Он работает как структурированный контекст для финального LLM-анализа:
- к текущему кейсу добавляются:
  - похожие документы (retrieval),
  - диагноз-специфичный паттерн.
- LLM выдает:
  - уровень риска,
  - возможные пропуски анамнеза,
  - возможные пропуски жалоб,
  - список уточняющих вопросов.

## 6. Критические риски и защитные меры

Риски:
1. Смещение паттерна к локальным практикам одного ЛПУ.
2. Недостаточная репрезентативность при малой когорте.
3. Галлюцинации LLM.
4. Избыточные «общие» рекомендации без клинической специфики.

Текущие меры:
1. Порог `minDocs`.
2. JSON-контракт результата.
3. Логирование fallback-сценариев.
4. Привязка паттерна к диагнозам + diagnosis-aware retrieval.

## 7. Почему выбран такой дизайн
1. Отделение pattern extraction от online-анализа уменьшает латентность анализа нового кейса.
2. Переиспользование паттерна снижает стоимость и нестабильность частых LLM-вызовов.
3. Наличие fallback поддерживает отказоустойчивость.
4. Хранение паттерна в БД обеспечивает аудит и повторяемость.

## 8. Что важно усилить дальше
1. Ввести TTL/ревизию паттернов (перегенерация при росте датасета).
2. Учитывать временную динамику документов (не только pooled-корпус).
3. Добавить контроль клинической валидности паттернов (экспертная обратная связь).
4. Вести quality-metrics паттерна (coverage, contradiction rate, alert precision proxy).
