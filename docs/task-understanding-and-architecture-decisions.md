# Понимание исходной задачи и основания архитектурных решений

## 1. Как я понял исходную задачу
Вы поставили задачу построить production-инфраструктуру для этапа подготовки данных и аналитики качества сбора анамнеза/жалоб:
1. получить из МИС обезличенные семплы документов,
2. нормализовать их,
3. систематизировать по диагнозам/комбинациям диагнозов,
4. сохранить в БД и в векторном индексе,
5. использовать ИИ для анализа нового текста и поиска вероятных пропусков в опросе пациента.

Ключевая бизнес-цель:
- снизить риск неполного сбора анамнеза/жалоб врачом при указанном диагнозе и сопутствующих диагнозах.

## 2. Важные вводные, которые напрямую повлияли на архитектуру

## 2.1. Технические ограничения
- платформа: `Node.js`;
- МИС на `Cache Object Script`;
- существующие процедуры править нельзя, можно писать новые;
- текущий этап: production-код для набора данных и систематизации;
- интеграционный дебаг с МИС вынесен в отдельный этап.

Следствие:
- добавлен новый COS routine (`cos-mis/MECRAG.int`), а не изменение старых процедур.

## 2.2. Интеграционные требования
- нужен код подключения к МИС,
- код подключения к OpenRouter,
- универсальный слой для LanceDB или sqlite-vec + SQLite,
- хранение результатов анализа.

Следствие:
- построены отдельные слои `mis / nlp / vector / ai / analysis / storage`.

## 2.3. Методологические требования
- выявление паттернов по диагнозам/совокупности диагнозов,
- референсы: MECR-RAG и Agentic Graph RAG,
- модельный стек: qwen 2.5 / qwen 3 через OpenRouter.

Следствие:
- реализован diagnosis-aware retrieval,
- добавлен Pattern layer,
- разделены модели OpenRouter для reasoning и embedding.

## 3. Какие источники контекста были использованы внутри репозитория
1. `mis/` (существующий Node MIS клиент): формат запросов, кодировка win1251, транспортные правила.
2. `cos-mis/IEPR.int`, `cos-mis/YIMH.int`, `cos-mis/YIMHH.int`: где хранятся документы, поля, индексы, доступные функции.
3. `cos-mis/AGENTS.md`: локальные конвенции и ограничения по COS-коду.
4. Ваши формулировки в задаче о целях анализа и границах текущего этапа.

## 4. Почему приняты именно такие архитектурные решения

## 4.1. Новый COS routine вместо модификации старых
Причина:
- прямое требование не менять существующие процедуры,
- безопаснее по риску регрессии в МИС.

Решение:
- `MECRAG.int` с отдельной точкой выгрузки обезличенных документов.

## 4.2. Потоковая выдача из COS через RWU
Причина:
- большие объемы выгрузки,
- риск переполнения/деградации при сборе большого JSON в одной переменной.

Решение:
- использование `OWF^RWU` + `W/WC^RWU` + `C^RWU`.

## 4.3. Универсальный векторный слой
Причина:
- требование поддерживать LanceDB или sqlite-vec/SQLite.

Решение:
- единый интерфейс vector store,
- конфиг-переключение `VECTOR_DB_ENGINE`.

## 4.4. Разделение OpenRouter моделей
Причина:
- разные профили нагрузки и качества у embedding и reasoning задач.

Решение:
- `OPENROUTER_EMBEDDING_MODEL` для `/embeddings`,
- `OPENROUTER_ANALYSIS_MODEL` для `/chat/completions`,
- `OPENROUTER_MODEL` как fallback.

## 4.5. Детерминированный контур + fallback
Причина:
- production-этап без интеграционного дебага требует устойчивости к внешним сбоям.

Решение:
- fallback-эмбеддинги,
- fallback-pattern,
- schema validation JSON-ответов,
- аудит в БД (`analysis_runs`).

## 4.6. Разделение по этапам (ingestion/pattern/analysis)
Причина:
- разные SLA, частота запуска и требования к трассируемости.

Решение:
- отдельные сервисы:
  - ingestion,
  - построение паттернов,
  - онлайн-анализ нового кейса.

## 5. Что сознательно НЕ делалось на этом этапе
1. Тонкая интеграционная настройка с конкретным production-MIS окружением.
2. Полный agentic graph orchestration (избыточно для этапа подготовки датасета).
3. Полноценная клиническая экспертная валидация паттернов и алертов.

## 6. Главные риски, принятые в текущем дизайне
1. Локальная специфика корпуса может искажать паттерны.
2. Regex-деидентификация ограничена.
3. Без экспертной разметки сложно измерить клиническую точность алертов.

## 7. Почему этот дизайн подходит под текущую стадию
1. Быстро запускается и дает сквозной рабочий контур.
2. Хорошо расширяется на следующий этап интеграционного дебага.
3. Сохраняет трассируемость и контролируемость каждого шага.
4. Не ломает существующую МИС-логику.
