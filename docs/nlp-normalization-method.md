# Метод NLP-нормализации

## 1. Назначение
Этот документ описывает production-метод нормализации медицинского текста для этапа подготовки датасета.
Цель: из разноформатных записей МИС получить стабильное, обезличенное и пригодное для retrieval/анализа представление.

## 2. Где используется в коде
- `src/nlp/normalize.js`
- `src/pipeline/datasetIngestionService.js`
- `src/analysis/caseAnalysisService.js`

## 3. Входные данные

### 3.1. Ретроспективный документ из МИС
Ожидаемые поля (минимум):
- `sample_id`
- `patient_ref`
- `doc_id`
- `doc_date`
- `doc_name`
- `diag_primary`
- `diag_all`
- секции текста: `complaints`, `anamnesis`, `findings` (или алиасы)

### 3.2. Новый кейс для анализа
- свободный текст (`text`) и/или поля `complaints`, `anamnesis`
- список диагнозов (опционально), если уже указан врачом

## 4. Пайплайн нормализации

## 4.1. Нормализация формата текста
Шаги:
1. унификация переносов строк (`\r\n` -> `\n`),
2. удаление неразрывных пробелов,
3. схлопывание повторяющихся пробелов,
4. ограничение пустых строк.

Зачем:
- уменьшает шум эмбеддингов,
- делает сравнение документов более устойчивым,
- уменьшает риск «ложной уникальности» текста.

## 4.2. Унификация разделов
Секции приводятся к единой структуре:
- `complaints`
- `anamnesis`
- `findings`
- `recommendation`

Даже если в источнике названия отличаются (`chief_complaints`, `history`, `objective`), внутри пайплайна используется единая схема.

Зачем:
- стабильный downstream-промпт,
- одинаковая структура для ретроспективы и нового кейса,
- проще проверять пропуски по конкретным разделам.

## 4.3. Деидентификация (дополнительная)
В тексте маскируются:
- телефоны -> `[PHONE]`
- SNILS-подобные шаблоны -> `[SNILS]`
- паспортоподобные шаблоны -> `[PASSPORT]`
- email -> `[EMAIL]`
- некоторые шаблоны ФИО/инициалов -> `[PERSON]`
- адресные фрагменты -> `[ADDRESS]`

Важно:
- это второй контур защиты поверх обезличивания на стороне МИС,
- деидентификация regex-уровня не гарантирует 100% удаление всех персональных данных,
- для прод-этапа рекомендуем дополнить словарной/NER-проверкой.

## 4.4. Извлечение и нормализация диагнозов
Используется регулярное извлечение ICD-подобных кодов из:
- явных полей диагноза,
- текста документа.

Правила:
- приведение к верхнему регистру,
- удаление дубликатов,
- сортировка,
- формирование `diagnosisKey` через `+`.

Пример:
- коды: `K74.6`, `K76.0`
- ключ: `K74.6+K76.0`

Зачем:
- diagnosis-aware retrieval,
- агрегирование паттернов на устойчивом ключе,
- трассируемость выборок.

## 4.5. Токенизация для статистики
Параллельно формируется облегченный токен-набор:
- lower-case,
- удаление пунктуации,
- фильтрация коротких токенов,
- исключение стоп-слов.

Зачем:
- быстрые эвристики,
- fallback-pattern без LLM,
- диагностика качества корпуса.

## 5. Выходной контракт
Нормализованный объект содержит:
- идентификаторы (`sampleId`, `sourceDocId`, `patientRef`),
- даты/имя документа,
- `diagnosisCodes`, `diagnosisKey`,
- структурированные `sections`,
- `text` (объединенный обезличенный текст),
- `tokens`,
- `metadata`.

Этот контракт сохраняется в SQLite и используется для построения эмбеддингов.

## 6. Контроль качества и отбраковка
Документ не идет в векторизацию, если:
- итоговый текст пуст,
- структура неконсистентна,
- ответ МИС невалиден.

Минимальные проверки в текущей версии:
- валидация формата входа,
- безопасный fallback при пустых полях,
- логирование проблемных страниц ingestion.

## 7. Почему выбран именно такой метод
1. Быстрый и воспроизводимый этап dataset prep.
2. Независимость от тяжелых NLP-моделей на этапе загрузки.
3. Низкий операционный риск: все шаги детерминированы.
4. Подготовка к последующему усилению (NER, клиническая онтология, temporal extraction) без смены базового контракта.

## 8. Ограничения текущей версии
- regex-деидентификация ограничена шаблонами,
- секционная сегментация не учитывает сложные полуструктурированные формы,
- извлечение ICD пока без контекстной валидации «ложноположительных» кодов,
- токенизация неглубокая (без лемматизации/морфологии).

## 9. Рекомендуемые усиления (следующий этап)
1. Добавить NER-деидентификацию и post-check на персональные сущности.
2. Добавить специализированные словари/правила по профилям (гепатология и др.).
3. Ввести quality-score нормализации по документу.
4. Добавить versioning нормализатора, чтобы переиндексация была контролируемой.
