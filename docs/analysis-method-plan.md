# План метода анализа (MECR-RAG style) + самокритика

## 1. Цель этапа
Собрать и систематизировать обезличенный корпус документов МИС для последующего контроля полноты сбора анамнеза и жалоб в новых кейсах.

## 2. Базовый метод (первичная версия)
1. Ingestion: выгрузка обезличенных документов из МИС порциями по датам и курсору.
2. Нормализация NLP: чистка шума, унификация разделов, извлечение ICD-10, дополнительная деидентификация.
3. Векторизация: эмбеддинги каждого документа.
4. Хранение:
   - SQLite: документ, секции, диагнозы, результаты анализа.
   - Vector DB (LanceDB или sqlite-vec/SQLite): быстрый similarity search.
5. Pattern layer: построение чек-листа ожидаемых вопросов по группе диагнозов из ретроспективы.
6. Анализ нового текста:
   - retrieval похожих кейсов + паттерна,
   - LLM-сравнение с текущим текстом,
   - вывод пропусков по анамнезу/жалобам и список рекомендуемых уточнений.

## 3. Проверка ошибок (критерии отказа)
1. Ошибки источника: пустой/битый ответ МИС, рассинхрон курсора.
2. Ошибки качества текста: пустой документ, невыделенные секции, шум в диагнозах.
3. Ошибки retrieval: мало релевантных кейсов при редком диагнозе.
4. Ошибки LLM: невалидный JSON, галлюцинации, клинически нерелевантные замечания.
5. Ошибки операционного уровня: таймаут OpenRouter, падение векторного слоя, mismatch размерности эмбеддинга.

## 4. Самокритика первичного метода
1. Риск data leakage: retrieval без жесткой фильтрации по диагнозам может добавлять нерелевантный контекст.
2. Риск overfitting к формулировкам: частотные паттерны из одного ЛПУ могут быть локальными, а не универсальными.
3. Риск ложных тревог: модель может маркировать "пропуск" там, где вопрос клинически не обязателен.
4. Риск недообнаружения: короткие или шаблонные записи уменьшают качество паттерна.
5. Риск непроверяемости: без структурного JSON-контракта анализ трудно аудировать.

## 5. Корректировки после критики
1. Добавлено: строгий JSON-контракт результата анализа + runtime-валидация схемы.
2. Добавлено: diagnosis-aware retrieval (фильтрация по пересечению ICD-кодов).
3. Добавлено: fallback-стратегии (локальные эмбеддинги, эвристический паттерн) вместо silent failure.
4. Добавлено: персистентный аудит (`analysis_runs`) с контекстом retrieval и моделью.
5. Исключено: кластеризация без метаданных диагноза на текущем этапе (высокий риск шума).
6. Исключено: fully-agentic многозвенный пайплайн на этапе dataset prep (лишняя сложность до стабилизации данных).

## 6. Что оставить на следующий этап
1. Clinical guardrails: отдельный rule-engine для hard-check красных флагов по специальностям.
2. Temporal reasoning: учет динамики во времени (а не только similarity по тексту).
3. Оценка качества: офлайн-метрики precision/recall на размеченном gold-set.
4. Human-in-the-loop: интерфейс подтверждения/отклонения замечаний модели.

## 7. Связь с референсами
Архитектура учитывает идеи MECR-RAG (медицинский retrieval + контекстная проверка) и Agentic Graph RAG (многошаговый контекст), но в текущем production-этапе сознательно ограничена deterministic pipeline для стабильности и трассируемости.
